# v0.4.5 — Dictionary Integrity + Per-Game Save Fix

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Fix two critical gameplay bugs — dictionary rejects valid word "LOONIE" and resuming a saved game loses board/rack state.

**Architecture:** Two independent fixes: (1) patch the bundled word list with missing SOWPODS/Collins words, (2) wire GameUI's autosave to the per-gameId key instead of the legacy key so each game retains its own state.

**Tech Stack:** TypeScript, Vite, Vitest, localStorage

---

## Root Cause Analysis

### Bug 1: Dictionary Validation

`src/data/wordList.ts` line 3787 shows `"loon","loonery","looney","loonier","loonies","looniest"` — **"loonie" is missing**. It has "looney" (alternate spelling) but not "loonie" (valid SOWPODS/Collins word, Canadian $1 coin). The dictionary was generated from a SOWPODS-derived source that may be missing ~year-2020+ additions. Other known gaps may exist.

### Bug 2: Per-Game Save

Flow trace:
1. `NewGameSetupPage.startGame()` → `createActiveGameRecord()` → metadata only → `navigate('game', {gameId})`
2. `showGame()` → no save found → creates `new Game(players)` → **saves to per-game key**: `GamePersistence.save(game, gameId)` → mounts GameUI
3. `GameUI.save()` at line 78-80 calls `GamePersistence.save(this.game)` — **no gameId** → saves to **legacy key** `openscrabble_save`
4. User makes a move, goes to hub → legacy key has new state, per-game key has initial state
5. User taps resume → `showGame()` calls `GamePersistence.load(gameId)` → loads **per-game key** → gets **old initial state** → board/rack are reset

**Fix:** GameUI must store and use its gameId when saving.

**Step 2: Update save() to use gameId**

```typescript
private save(): void {
  if (this.gameId) {
    GamePersistence.save(this.game, this.gameId);
  } else {
    GamePersistence.save(this.game);  // fallback to legacy
  }
  if (this.onAutoSave) {
    const state = this.game.getState();
    this.onAutoSave(
      state.players.map((p) => p.score),
      state.turnNumber
    );
  }
}
```

## Risks and Open Questions

- **Dictionary completeness**: The MISSING_WORDS array is a manual patch. If more missing words are discovered during QA, add them to the array.
- **Backward compatibility**: Existing legacy `openscrabble_save` key is preserved. GameUI saves to per-game key when gameId is provided, falls back to legacy key when not.
- **GameID lifecycle**: The gameId is created in NewGameSetupPage and passed through main.ts → showGame → GameUI. If GameUI is constructed without a gameId (e.g., from a legacy code path), it falls back to the legacy key. This is safe.
- **localStorage limits**: Each active game stores a full game state (~5-15 KB). With 100-record cap in LocalGameStore, the worst case is ~1.5 MB, well within localStorage's 5-10 MB limit.
